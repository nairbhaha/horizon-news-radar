Microsoft's Majorana-based topological approach has been one of the most debated topics in quantum computing, and a rigorous scientific challenge to these claims has major implications for the direction of quantum hardware research and the credibility of topological quantum computing as a viable path to fault-tolerant quantum computers. The Majorana 2 chip is built on the concept of Majorana zero modes, which are quasiparticle excitations that behave as their own antiparticles and are theorized to provide topological protection against decoherence, but the formal critique argues that Microsoft's evidence for these modes remains insufficient.

rss · Decrypt · Jun 24, 19:07

**Background**: Topological qubits are a proposed type of quantum bit that stores quantum information in the topological properties of a physical system rather than in individual particles, making them theoretically far more stable against errors than conventional qubits. Majorana zero modes, first hypothesized by physicist Ettore Majorana in 1937, are exotic quasiparticles that can emerge in certain semiconductor-superconductor hybrid systems at temperatures near absolute zero under strong magnetic fields. Microsoft's approach aims to use these Majorana-based topological qubits to achieve fault-tolerant quantum computing, but the field has faced repeated scrutiny, including a high-profile retraction of a 2018 Nature paper that had claimed evidence of Majorana zero modes.

## [Django 6.0 Introduces Built-in Tasks Framework](https://realpython.com/django-tasks/) ⭐️ 8.0/10

Django 6.0 has introduced a new built-in tasks framework that allows developers to define background jobs using the @task decorator, enqueue work, check task results, and use named queues. Published on May 12, 2026, the Real Python tutorial walks through this first-party Django-native approach to background work. This is a major shift for the Python web ecosystem because it provides a lightweight, first-party alternative to heavyweight solutions like Celery, lowering the barrier for Django developers who need background job processing. It could reduce dependency on third-party task queue libraries and standardize how Django projects handle asynchronous work. The framework is not a queue, worker system, or scheduler itself — it only defines background work in a Django-native way and provides hooks for external systems to execute that work. By convention, it looks for a tasks module in each installed app, and the @task decorator wraps functions in a Task object that exposes .enqueue() and related methods for sending work to a worker.

rss · Real Python · Jun 24, 14:00

**Background**: Django is one of the most popular Python web frameworks, widely used for building web applications. Historically, Django developers who needed to run background jobs — such as sending emails, processing images, or generating reports — relied on third-party task queue systems like Celery, Django Q, or django-background-tasks. These solutions typically require additional infrastructure such as Redis or RabbitMQ as message brokers, and separate worker processes to consume and execute queued tasks. The new Django Tasks framework changes this landscape by providing a native abstraction layer for defining background work without mandating a specific execution backend.

A technical exploration reveals that running database drivers as sandboxed external processes can significantly reduce attack surfaces, with the unexpected insight that a simple HashMap insert operation can serve as a critical security boundary between trusted and untrusted code. This approach challenges conventional thinking about where trust boundaries exist in database connectivity layers, potentially influencing how systems programmers and security engineers design database drivers to minimize vulnerabilities from compromised or malicious drivers. The article discusses how database drivers are traditionally loaded into the application's memory space via APIs like LoadLibrary on Windows, and proposes running them as separate sandboxed processes instead, with the HashMap insert operation acting as a serialization point that enforces a trust boundary between the application and the untrusted driver process.

**Background**: Database drivers are software components that enable applications to communicate with database management systems through standard interfaces like ODBC (Open Database Connectivity). Sandboxing is a security technique that runs code in an isolated environment with restricted permissions, limiting the potential damage from malicious or vulnerable code. A HashMap is a data structure that stores key-value pairs, and in this context, the act of inserting data into a HashMap serves as a controlled point where data crosses from an untrusted external process back into the trusted application.
